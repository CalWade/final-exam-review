1 Internet 协议栈与 OSI 参考模型 🎯 真题命中

Q (简答): Describe the layers of the Internet protocol stack and list one protocol in each layer.
答案: 5 层（Application / Transport / Network / Link / Physical），每层举例 HTTP、TCP、IP、Ethernet、10Base-T。
关键词 protocol stack layer application transport network link physical
为什么出现

早期网络各厂商各自为政（IBM SNA、DEC DECnet），互不兼容。1974 年 IBM 提出 SNA，1977 年 ISO 开始制定 OSI，核心思想都是分层——把复杂的网络通信拆成若干层，每层只关心自己的事，层与层之间通过接口交互。

  • 每层可以独立设计和替换（换掉物理层从电缆变光纤，上层无感知）
  • 标准化后不同厂商可以互操作
  • 问题定位更容易（"是物理层断线还是网络层路由错误？"）
实际应用
  • 排障：运维说"Layer 1 故障"= 网线松了；"Layer 3 故障"= 路由配置错了
  • 开发：写 Web 应用只关心应用层（HTTP），不用管 TCP 怎么传
  • 安全：防火墙在 Layer 3/4 过滤，WAF 在 Layer 7 过滤

OSI 在工程中的"实际变形"

QUIC 打破了传统分层：HTTP/3 的底座 QUIC 运行在 UDP 之上，但在 UDP 之上重建了 TLS、流量控制、拥塞控制——本质上是把传输层和安全层的功能搬到了应用层，绕过内核和中间设备的固化行为。Chrome、Edge 都用 QUIC，YouTube 视频 70%+ 走 QUIC。

SDN 与 Service Mesh 重塑网络层：Service Mesh（Istio + Envoy）在应用层之上构建"通信基础设施层"——OSI 模型之外的新层次，承担负载均衡、熔断、重试、流量染色。

大厂面试延伸

为什么 TCP/IP 模型比 OSI 在工程上更成功？
OSI 过于理想化（会话/表示层实际很少单独存在），实现复杂、效率低；TCP/IP 来自 ARPANET 工程实践，"先有实现后写规范"（rough consensus and running code），更贴合实际。
浏览器 Network 面板 Timing 标签对应哪些层？
Queueing（应用调度）→ DNS（应用层）→ Initial connection（TCP 传输层）→ SSL（原表示层）→ Request sent（应用层）→ Waiting TTFB（服务器+网络）→ Content Download（数据传输）。
OSI 协议族本身（FTAM/VT/MOTIS）被 TCP/IP 体系的 FTP/Telnet/SMTP 取代；NetBIOS/NetBEUI 被 TCP/IP 上的 SMB 取代；X.25 被 IP 数据报方式取代；ATM 被 IP+Ethernet 打败。

2 物理链路类型——导引与非导引媒介 🎯 真题命中

Q (单选): Which of the following is a guided transmission medium?
答案: A. Twisted-pair cable（双绞线）
导引型 = 有线 = 双绞线/同轴/光纤；非导引型 = 无线 = WiFi/卫星/微波。
关键词 guided unguided twisted-pair medium
为什么区分

信号需要介质传播。导引型是"看得见摸得着"的线缆（双绞线、同轴、光纤），非导引型是电磁波在自由空间传播（无线电、微波、红外）。选择取决于距离、成本、环境、带宽需求。

实际应用
  • 导引型：机房互联（光纤）、家庭上网（双绞线/FTTH）、有线电视（同轴）
  • 非导引型：手机上网（4G/5G）、卫星通信（Starlink）、蓝牙、Wi-Fi

为什么家用 Wi-Fi 5GHz 比 2.4GHz 速率高但穿墙弱？
频率高→可用带宽更大；频率高→路径损耗大、穿墙弱，但近距离吞吐量高。Wi-Fi 6E 开放 6GHz 频段同理。
跨太平洋海底光缆的延迟能更低吗？
上海到美西约 12000km，光在光纤中传播速度 2×10⁸ m/s（光速 2/3），单程 ~60ms，往返 ~120ms。这个延迟由物理决定，无法突破——除非量子通信（量子纠缠本身不传输信息）。

3 节点的四种时延 🎯 真题命中

Q (单选): For packet-switched network, L=6Mbits, R=3Mbps, the transmission delay from A to B is?
答案: A. 2 秒   公式 d_trans = L/R = 6M/3M = 2s
关键词 transmission delay packet-switched L/R
为什么理解时延

网络性能核心指标。"网速慢"到底是哪里慢？是发送慢、排队慢、传播慢还是处理慢？理解四种时延才能对症下药。

深度讲解
图 3.1 四种时延发生位置
时延定义公式决定因素
处理时延检查头部、决定转发—硬件性能（μs 级）
排队时延输出队列等待ρ = La/R网络拥塞程度（变化最大）
传输时延比特推上链路L / R分组大小 + 链路带宽
传播时延比特在介质中跑d / s物理距离 + 传播速度
传输时延 vs 传播时延（高频考点！）
传输时延 = 把水倒进管道的时间（取决于桶大小和倒的速度）
传播时延 = 水在管道中流动的时间（取决于管道长度和流速）
带宽再高也解决不了传播时延！100Mbps 链路传 1bit 传输时延 10ns，传播时延可能 20ms——差百万倍。
排队时延的非线性：ρ → 0 几乎无排队；ρ → 1 排队时延趋于无穷；ρ > 1 队列溢出丢包。这就是限流存在的根本原因。
🌐 现代技术栈演进

传播时延 → CDN 与边缘计算：用户从北京访问美国 RTT 120ms+，CDN 把资源缓存到全国节点，传播时延降到 < 20ms。HTTP/3 over QUIC 把 TCP+TLS 握手合并成 1 RTT。

排队时延 → 限流 + 负载均衡 + AQM：Sentinel 网关限流、LVS/Nginx 负载均衡、fq_codel/CAKE 对抗 Bufferbloat。

处理时延 → 硬件加速：ASIC + TCAM 实现线速转发；DPDK/kernel bypass 跳过内核协议栈。

服务器（或初始种子）必须至少把完整文件的每个 chunk 都上传到网络中一次，否则其他节点永远无法拼出完整文件。P2P 的优势是后续分发由节点承担，但最初的完整文件源不可省略。

5 HTTP 非持久连接访问网页的过程 🎯 真题命中（高频）

Q1 (单选): Non-persistent connection HTTP, how long does it take to send an object?
答案: B. 2RTT + transmit time
Q2 (简答): Describe the process of accessing a web page with a non-persistent HTTP.
答案: ① TCP 三次握手（1 RTT）→ ② HTTP 请求 + 响应（1 RTT + 传输时间）→ ③ 连接关闭。总耗时 2RTT + transmit time。
关键词 non-persistent RTT TCP handshake transmission delay
为什么有非持久连接

HTTP/1.0 默认非持久（短连接）。最初网页简单——一个 HTML 文件传完就断，省资源。后来网页越来越复杂（图片、CSS、JS），每次重连 TCP 太浪费，所以 HTTP/1.1 引入了持久连接。

非持久连接的四大缺点

  1. 每个对象都要新建 TCP 连接——三次握手开销大
  2. TCP 慢启动浪费——刚加速就断了
  3. 服务器负担重——每对象占一套 TCP 状态

HTTP 协议演进谱系

HTTP/0.9 (1991) → HTTP/1.0 短连接 → HTTP/1.1 持久+流水线 → HTTP/2 多路复用 → HTTP/3 over QUIC
HTTP/1.0 时代如何优化页面加载？
① 域名分片（绕过单域名 6 连接限制）；② 雪碧图合并小图标；③ Data URI 内联；④ 合并 JS/CSS。这些技巧在 HTTP/2 时代过时了。
短连接在大规模服务中的隐藏成本？
① TIME_WAIT 堆积——主动关闭方等 2MSL（60-120s），高 QPS 快速耗尽端口；② TCP 状态消耗内存；③ 三次握手耗 CPU；④ TLS 握手成本巨大。现代服务都用连接池 + 持久连接。
HTTP/0.9 无现代应用；HTTP/1.0 默认短连接在主流 Web 已不用；域名分片在 HTTP/2 时代有害无益；雪碧图被多路复用取代；Data URI 大资源内联应避免。

6 HTTP 持久连接访问网页的过程 🎯 真题命中

Q1 (判断): Web cache can reduce link traffic, but cannot reduce access time.   F (错误)
Q2 (简答·套二): Briefly describe the purpose of using proxy server and what are the benefits?
答案: 目的：代理服务器代表原始服务器满足 HTTP 请求，缓存对象副本。
好处：① 缩短响应时间；② 减少链路带宽；③ 减轻服务器负载。
关键词 persistent web cache proxy server response time
为什么需要持久连接

非持久连接每对象重连 TCP，开销巨大。现代网页平均 50-100 个对象，如果每个都重新握手，体验极差。HTTP/1.1 引入持久连接（长连接），一次 TCP 连接可传多个对象。

深度讲解

三种方式对比

方式N 个对象总 RTTTCP 连接数
非持久（串行）2N RTTN
持久 + 非流水线(N+1) RTT1
持久 + 流水线≈ 2 RTT1
流水线方式：客户端连续发送多个请求，不必等前一个响应。理想情况下 N 个对象只需 2 RTT（1 握手 + 1 请求/响应）。HTTP/1.1 默认方式。
但 HTTP/1.1 流水线没大规模部署——队头阻塞（前一个响应慢后面全堵）+ 中间代理兼容性差。浏览器改用 6 个并行 TCP 连接，每连接串行处理。

10 GBN（回退 N 步）与选择重传（SR） 🎯 真题命中

Q1 (判断): For go-back N, ack(n) represents that n and previous packets of n have been received correctly.   T (正确) — 累积确认 (Cumulative ACK)
Q2 (简答·套二): Please briefly describe selective repeat (SR).
答案: SR 仅重传出错分组；接收方逐个单独 ACK；发送方每分组独立定时器；接收方缓存失序分组；窗口 ≤ 序号空间一半。
关键词 go-back-N cumulative ACK selective repeat individual ACK window size
为什么需要这两个协议

在不可靠信道上实现可靠传输，核心问题：① 如何处理丢包？② 如何高效利用带宽？最简单的停等协议每发一个包等确认，效率极低（信道利用率 < 1%）。流水线协议连续发多个包再等确认。GBN 和 SR 是两种不同策略。

深度讲解

GBN（Go-Back-N）核心思想

发送方连续发 N 个分组，但接收方只按序接收。一旦某分组丢失，从该分组开始全部重传。

  • 累计确认：ACK(n) 表示 n 之前所有分组已收到
  • 一个全局定时器，超时重传所有未确认分组
  • 接收方丢弃乱序分组

SR（Selective Repeat）核心思想

接收方缓存乱序到达的分组，发送方只重传真正丢失的分组。

  • 每个分组独立定时器
  • 逐个确认
  • 接收窗口也缓存

特性GBNSR

定时器一个全局每个独立
窗口限制序号空间 ≥ N+1序号空间 ≥ 2N

SR 窗口大小限制：发送窗口 + 接收窗口 ≤ 序号空间，即 2N ≤ 序号空间，N ≤ 序号空间/2。

TCP 实际是 GBN + SR + SACK 的混合体

教材里 GBN 和 SR 是独立协议，真实 TCP 是两者的混合 + 工程优化：

  • 累计确认（类 GBN）：ACK(n) 表示 n 之前所有收到
  • 不丢弃乱序（类 SR）：接收方缓存乱序段
  • SACK 选择确认：精确告知"我已收到哪些不连续块"
  • 3 个重复 ACK 触发快重传

现代可靠传输的工程模式

  • NACK + 重传：发现空洞主动发 NACK，比超时快（WebRTC、卫星）
  • FEC 前向纠错：发送冗余包，丢包时恢复无需重传（Reed-Solomon、LDPC、Polar）
  • 视频流可错性分级：I 帧必须可靠，P/B 帧可丢弃
TCP 选择累计确认而非逐个确认的原因？
① 简洁——ACK 一个序号代表前面所有收到；② 容错——即使 ACK 丢失，后续 ACK 也能确认；③ 兼容 SACK——SACK 作为选项补充精确信息。
5G 为什么要用 LDPC 和 Polar 码？
① 香农极限——LDPC 长码块接近极限，Polar 短码块接近；② 性能——比 4G Turbo 码提升 1-2dB；③ 华为推动 Polar 进入 5G 控制信道，高通推动 LDPC 进入数据信道。中国通信厂商首次在国际标准中拥有重要话语权。
纯 Stop-and-Wait 效率太低；纯 GBN 不缓存乱序浪费带宽；TCP Reno 被 NewReno/Cubic/BBR 取代；传统 ARQ 无 FEC 在卫星/深空通信被 Hybrid ARQ 取代。

12 TCP 流量控制 🎯 真题命中

Q (判断): For the receiver of TCP, when a packet arrives in sequence and all previous packets have responded to an ACK, the receiver immediately replies to the ACK for that packet.
答案: F (错误) — TCP 使用延迟确认 (Delayed ACK)，按序到达不会立即回复，而是延迟 ≤ 500ms 期望捎带或合并。
关键词 delayed ACK piggybacking in sequence flow control
为什么需要流量控制

发送方可能很快，接收方可能很慢。如果不管不顾地发，接收方缓冲区就溢出了——数据丢失。流量控制让发送方"看接收方的脸色行事"，确保接收方来得及处理。

深度讲解

核心机制：滑动窗口

TCP 接收方在每次 ACK 中通过"窗口大小"（rwnd）字段告诉发送方："我的缓冲区还能接收多少字节"。

rwnd = RcvBuffer - 已缓存数据   |   发送方保证：在途数据量 ≤ rwnd
图 12.1 TCP 发送方滑动窗口

零窗口问题

当 rwnd=0 时，发送方必须停止发送。但接收方腾出空间后的通知可能丢失，导致死锁。解决方案：持续计时器（Persist Timer），发送方定期发零窗口探测报文（1 字节），强制接收方回复当前窗口。

流量控制 vs 拥塞控制：
流量控制保护接收方（rwnd），拥塞控制保护网络（cwnd）。
实际发送窗口 = min(rwnd, cwnd)。
流量控制是接收方告诉发送方"我处理不过来"，拥塞控制是发送方自己判断"网络拥塞了"。

🎯 真题考点：TCP 延迟确认（Delayed ACK）

规则：当按序报文到达、且之前的报文都已确认时，接收方不会立即回复 ACK，而是延迟一段时间（通常 ≤ 500ms，Linux 默认 40ms）。
目的：① 期望有反向数据可以捎带确认（Piggybacking）；② 等待下一个按序报文到达后合并确认，减少 ACK 报文数量。
真题陷阱：题目"按序到达且之前都已 ACK，立即回复 ACK"——错误，TCP 用 Delayed ACK，不会立即回复。
触发立即 ACK 的条件：① 失序报文（立即发重复 ACK，触发重传）；② 收到报文填补了缺口；③ 延迟计时器到期。
🌐 现代技术栈演进

窗口缩放（Window Scaling）

原始 TCP 头部窗口字段 16 位，最大 64KB。在 100Mbps 链路上，RTT 必须 < 5ms 才能跑满带宽——跨地区不行。窗口缩放因子 S（0-14），实际窗口 = 头部值 × 2^S，最大 1GB。Linux 默认缩放因子 7（窗口 8MB）。

net.ipv4.tcp_moderate_rcvbuf = 1  # 自动调优

糊涂窗口综合征（SWS）

接收方缓慢读取数据，每次通告小窗口（如 1 字节），发送方每次发小包——头部开销远大于数据。解决：① D.Clark 算法——窗口小于 MSS 不通告；② Nagle 算法——小数据积累到 MSS 再发。实时应用应关闭 Nagle（TCP_NODELAY）。

分布式系统的"背压"（Backpressure）

流量控制思想在分布式系统中广泛应用：

  • 消息队列：消费者慢时 broker 减速
  • 反应式编程：RxJava onBackpressureBuffer
  • gRPC 流式：HTTP/2 流级流控
  • 微服务网关：Sentinel、Hystrix 背压
  • 数据库连接池：HikariCP 满时阻塞
Nagle 算法默认开启在实时应用被 TCP_NODELAY 关闭；固定缓冲区大小被自动调优取代；基于 rwnd 的死板流控被 HTTP/2/QUIC 的基于信用流控取代。

13 TCP 连接建立及关闭

为什么三次握手四次挥手

TCP 是全双工——双方都可独立发送和接收。建立连接时要确保双方都准备好；关闭连接时要确保双方都发完了数据。

深度讲解

TCP 三次握手

图 13.1 TCP 三次握手
为什么三次？两次不行吗？
两次握手无法防止"已失效的连接请求"。客户端发 SYN 被延迟，超时重发 SYN 建连完成数据传输关闭，旧 SYN 才到达服务器。若两次握手，服务器直接建连浪费资源。三次握手中，客户端不会对旧 SYN 的回复发 ACK，连接不会真正建立。

TCP 四次挥手

图 13.2 TCP 四次挥手 + TIME_WAIT
为什么四次？ TCP 全双工，FIN 只表示"我不再发数据"，但对方可能还有数据要发。
② ACK 确认收到 FIN，但服务器还能继续发；③ FIN 服务器也发完了才请求关闭；②和③之间可能有数据传输，不能合并。

TIME_WAIT = 2MSL：① 确保最后 ACK 到达（丢失则服务器重发 FIN）；② 让本连接报文消亡，避免新连接收到旧报文。

SYN Flood 攻击与 SYN Cookies

攻击者伪造源 IP 大量发 SYN，服务器分配资源回复 SYN+ACK 后等 ACK（半开连接）。半开连接占满 SYN 队列。

SYN Cookies（Bernstein 1996）：服务器不保存半开连接状态，把状态编码进 SYN+ACK 的 ISN。客户端 ACK 时携带 ISN+1，服务器解码验证。核心思想：把存储换算成计算。

TCP Fast Open（TFO）

首次握手 1-RTT，后续连接 0-RTT 数据传输。Linux 3.6+ 支持，Chrome、Curl 启用。

大厂最佳实践：服务端让客户端主动关闭，把 TIME_WAIT 转嫁给客户端；客户端用连接池复用 TCP。

服务器大量 CLOSE_WAIT 是什么原因？
CLOSE_WAIT 是被动关闭方收到 FIN、回复 ACK 后等待应用 close()。大量 CLOSE_WAIT 说明应用代码没正确关闭连接——可能忘了 close，或代码 bug 导致 socket 泄漏。
为什么 TIME_WAIT 是 2MSL 而不是 1MSL？
① 1 MSL 等待最后一个 ACK 到达服务器（最坏 ACK 在路上时刚好丢失）；② 另 1 MSL 等待服务器重发的 FIN 到达。两个 MSL 覆盖"ACK 丢失 + FIN 重传"的最坏场景。
HTTP/3 (QUIC) 的握手为什么能 0-RTT？
① QUIC 把 TCP 握手和 TLS 握手合并——首次 1-RTT；② 重连时用之前协商的密钥发 0-RTT 数据包；③ 连接迁移用 Connection ID，IP 变化不需重新握手。
tcp_tw_recycle 在 Linux 4.12 移除（NAT 环境问题）；TFO 大规模部署因兼容性问题被 HTTP/3 的 0-RTT 取代；传统三次握手在 QUIC 时代被 1-RTT/0-RTT 取代。

16 网络层数据包分片及计算；MTU、MSS 含义

为什么需要分片

不同链路有不同 MTU。以太网 MTU=1500，PPPoE 是 1492。当 IP 数据报大于链路 MTU 时，路由器必须分片，每个片独立传输，在目的端重新组装。

深度讲解

MTU 与 MSS

MTU（最大传输单元）：链路层帧承载的最大数据量 = 1500 字节（以太网），包含 IP 头 + IP 数据
MSS（最大报文段长度）：TCP 数据部分最大长度 = MTU - IP头(20) - TCP头(20) = 1460 字节
关系：MSS 是 TCP 为避免 IP 分片而主动限制的发送大小。

IP 分片相关字段

  • 标识（Identification）：同一数据报的所有分片标识相同
  • 标志（Flags）：MF=1 后面还有分片；DF=1 不允许分片
  • 片偏移（Fragment Offset）：以 8 字节为单位

"绝不在网络中分片"哲学

现代网络工程核心原则：避免分片。

  • Path MTU Discovery（PMTUD）：发送方设 DF=1，路由器无法转发时返回 ICMP "Fragmentation Needed"，发送方降低 MSS
  • PMTU 黑洞：ICMP 被防火墙拦截，发送方收不到反馈，连接卡死
  • PLPMTUD：不依赖 ICMP，靠传输层探测
  • TCP MSS Clamping：握手时改小 MSS（PPPoE 1452、VXLAN 1450）

用户反馈"HTTPS 网站打不开，HTTP 可以"，可能是什么原因？

为什么需要 SDN

传统路由器控制平面和数据平面紧耦合——每台路由器独立运行路由算法。导致：① 管理困难；② 创新困难；③ 不一致。SDN 核心思想：把控制平面从路由器中分离出来，集中到控制器上，路由器只负责按指令转发。

DHCP 安全：被忽视的攻击面

  • 饿死攻击：攻击者伪装大量 MAC 发 DHCPDISCOVER 耗尽 IP 池。防御：DHCP Snooping 限制速率
  • 伪 DHCP 服务器：劫持流量。防御：DHCP Snooping 标记"信任端口"

DHCPv6（有状态）：服务器记录地址分配，可下发 DNS 等。
SLAAC + DHCPv6：SLAAC 分地址，DHCPv6 下发附加配置。移动网络常用。

802.1X：基于端口的网络接入控制。客户端 supplicant → 交换机 authenticator → RADIUS 服务器。RADIUS 后端通常对接 LDAP / Active Directory 做用户身份查询——LDAP 是企业身份目录事实标准。微软 AD = LDAP + Kerberos + Group Policy。Cisco ISE、Aruba ClearPass、华为 Agile Controller 都内置 LDAP Connector。

为什么 DHCP 用 UDP 而不是 TCP？
① 客户端没有 IP 时无法建立 TCP 连接（TCP 需要源 IP）；② UDP 广播简单高效；③ DHCP 一次交互就完成，无需可靠传输——丢失则重发；④ 减少开销。
云服务器重启后为什么 IP 不变？
云厂商通过静态 IP 绑定实现——VPC 中 IP 与虚拟机实例绑定（在 metadata 中持久化），每次启动 hypervisor 注入相同 IP。即使重启、迁移，IP 不变。

Q1 (单选·套一): What is the checksum of E663 and D555?   D. BBB9 (回卷和)
Q2 (单选·套二): What is the checksum of E666 and D555?   D. BBBC (回卷和)
Q3 (综合题3-2): D=1011011, G=1001, give the CRC code.   R = 011
算法：十六进制加法 → 进位回卷 → 取反得最终校验和。
关键词 checksum one's complement sum wrap-around CRC generator mod-2 division XOR
为什么需要 CRC

数据在链路上传输时可能因噪声、干扰出错。简单方法（奇偶校验）只能检测奇数个错误。CRC 能检测多位错误，广泛用于链路层。

深度讲解

CRC 计算原理

CRC 将数据视为多项式，通过模 2 除法计算余数。

  • D（Data）：要发送的数据，d 位
  • G（Generator）：生成多项式，r+1 位（最高位和最低位必须为 1）
  • R（Remainder）：CRC 校验码，r 位
发送方：D 左移 r 位 → 对 G 做模 2 除法 → 得余数 R → 发送 D×2^r ⊕ R
接收方：用收到的数据对 G 做模 2 除法，余数 0 无差错，非 0 有差错。
模 2 除法：与普通除法的区别——减法用异或（XOR）代替，不借位。

计算示例

数据 D = 101110，生成多项式 G = 1001（r=3）

第1步：D 左移 3 位 → 101110000

第2步：模 2 除法

  101110000
  1001          ← 第1次异或
  ------
  001010000     ← 余数，去掉前导0
    10100
    1001        ← 第2次异或
    -----
     001100
       1100
       1001     ← 第3次异或
       ----
        1010
        1001    ← 第4次异或
        ----
         011    ← 余数 R = 011

第3步：发送 101110011（原数据 + 校验码）

CRC 检错能力

  • 能检测所有单比特错误
  • 能检测所有双比特错误（如果 G 含 (x+1) 因子）
  • 能检测所有奇数个错误（如果 G 含 (x+1) 因子）
  • 能检测所有长度 ≤ r 的突发错误
  • 能检测大部分长度 > r 的突发错误（概率 1 - 2^(-r)）

🎯 真题考点：Internet Checksum（因特网校验和）

出现位置：IP / TCP / UDP 头部（不仅是 UDP 头部那 2 字节字段，而是整套算法）。
核心思想：把数据按 16 位（2 字节）为一组进行反码求和（One's Complement Sum），最高位进位回卷（Wrap-around）加到最低位，最后对结果按位取反得到校验和。
为什么用反码：发送方与接收方用同一套运算，接收方算完全 0xFFFF 即无错；发送方把校验和字段置 0 后参与运算，让"含校验和在内的反码和"取反恰为 0，运算对称、硬件实现简单（end-around-carry 加法器）。

计算步骤（必考）

数据按 16 位分组 → 反码求和（含回卷） → 结果取反 → 得 Checksum
  • 反码求和：普通二进制加法 + 把最高位进位"回卷"加回低位（不是丢掉！）
  • 回卷规则：若求和结果超出 16 位（产生第 17 位进位 1），则把 1 加到低 16 位
  • 最终取反：把回卷后结果的每一位 0↔1 翻转，得到 Checksum 字段值
  • 接收方校验：把含校验和在内的所有 16 位字反码求和，全 1（0xFFFF）即无错

Checksum vs CRC 对比？
① 运算单位——Checksum 16 位字加法，CRC 比特级模 2 除法；② 检错能力——CRC-32 远强于 Checksum（Checksum 仅检测少数错误模式）；③ 速度——Checksum 软件实现快（16 位加法），CRC 需查表或硬件；④ 应用——Checksum 用于 IP/TCP/UDP 头部（端到端），CRC 用于链路层帧（每跳重算）；⑤ 现代趋势——IPv6 取消了 IP 头部 Checksum（交给上层 TCP/UDP），但 TCP/UDP 仍保留。

为什么需要随机访问协议

多个设备共享同一条广播链路（如同轴电缆、无线信道），如果同时发送就会冲突。随机访问协议规定了谁可以发、什么时候发、冲突了怎么办。

纯 ALOHA vs 时隙 ALOHA

纯 ALOHA：想发就发，冲突后等随机时间重发。易受冲突时间 2×T_fr。最大效率 1/(2e) ≈ 18.4%。

时隙 ALOHA：把时间分成等长时隙，只能在时隙开始时发。易受冲突时间 1×T_fr。最大效率 1/e ≈ 36.8%。效率翻倍但需要时间同步。

CSMA 三种坚持策略

  • 1-坚持：持续侦听，一空闲就发（冲突概率高）
  • 非坚持：发现忙就等随机时间再侦听（减少冲突但延迟增加）
  • p-坚持：空闲时以概率 p 发送

CSMA/CA（Wi-Fi 用）

为什么不用 CSMA/CD？ 无线环境中很难检测冲突（隐蔽站问题），且发送时无法同时准确接收检测。

规则：发送前侦听，空闲等待 DIFS 后发送；接收方正确收到后等待 SIFS 发 ACK；发送方收到 ACK 成功，未收到重发。RTS/CTS 握手解决隐蔽站问题。

Wi-Fi 6/7 的 OFDMA

不再用纯 CSMA/CA（一个时刻一个用户发送）。把信道分成多个子载波（RU），不同用户用不同 RU 同时传输。类似"4 车道高速公路"，多个用户并行。机场、商场等密集场景吞吐量提升 4 倍。

5G NR 的多址接入

  • OFDMA：下行，不同用户不同子载波
  • SC-FDMA：上行，峰均比低
  • 基站集中调度，无随机访问冲突
  • 随机接入（RA）：仅初始接入时用，类似时隙 ALOHA

LP-WAN（低功耗广域网）

  • LoRa：CSS 扩频，多用户同时同频不冲突
  • NB-IoT：基于 LTE 的 OFDMA
  • Sigfox：超窄带，UNB
现代交换式以太网为什么不用 CSMA/CD？
① 交换机每端口独占介质，无冲突；② 全双工通信，发送接收独立；③ CSMA/CD 的"边发边听"无意义。CSMA/CD 仅在集线器和同轴电缆以太网中需要，现代以太网已淘汰。
Wi-Fi 6 的 OFDMA 为什么提升密集场景性能？
① 多用户并行——传统一个时刻一个用户传；② RU 调度——小数据用户小 RU，大数据用户大 RU；③ 减少退避；④ 低延迟——小包用小 RU。
纯 ALOHA 仅作理论；时隙 ALOHA 在早期卫星通信用被 DAMA/TDMA 取代；CSMA/CD 在交换式以太网退场；集线器（Hub）被交换机取代；RTS/CTS 在现代 Wi-Fi 默认禁用。

23 CSMA/CD 算法 🎯 真题命中

Q (简答): Please briefly describe work process for Ethernet CSMA/CD algorithm.
答案: ① CS 载波监听：发送前侦听信道；② CD 冲突检测：边发边听，发现冲突立即停发；③ Jamming：发拥挤信号；④ Exponential Backoff：随机退避后重传。
关键词 CSMA/CD carrier sense collision detection jamming exponential backoff
深度讲解

CSMA/CD 算法流程

侦听信道 → 空闲则发送 → 边发边听 → 冲突停发 → 二进制指数退避 → 重发

关键参数

2τ（争用期/冲突窗口） = 往返传播延迟
τ = 信号在总线上单程传播最大延迟

最小帧长度：发送时间必须 ≥ 2τ，否则发完了才发现冲突
以太网最小帧 = 64 字节（数据最少 46 字节，不足填充 Padding）
10Mbps 以太网：2τ = 51.2μs → 512 bit = 64 字节

二进制指数退避

  • 第 1 次冲突：从 {0, 1} 选 k，等待 k×2τ
  • 第 2 次冲突：从 {0, 1, 2, 3} 选 k
  • 第 n 次冲突：从 {0, 1, ..., 2^min(n,10) - 1} 选 k
  • 最多重试 16 次，超过放弃

CSMA/CD 效率

效率 = 1 / (1 + 5×τ_prop / τ_trans)

当帧很长时（τ_trans >> τ_prop），效率接近 1。

🌐 现代技术栈演进

CSMA/CD 在现代以太网的"退场"

关键事实：现代以太网（交换式 + 全双工）几乎不再使用 CSMA/CD。

同轴电缆 → 集线器+双绞线 → 交换机+全双工 → CSMA/CD 退场

CSMA/CD 退场后的流量管理

  • PAUSE 帧（802.3x）：接收方缓冲快满时让对端暂停
  • PFC 优先级流控（802.1Qbb）：8 个优先级独立流控
  • ECN 显式拥塞通知：IP 头部 2 位标志
  • QCN 量化拥塞通知（802.1Qau）：L2 拥塞通知

无丢包网络（Lossless Network）

通过 PFC + ECN + DCBX 实现。用于 RoCE（RDMA 不重传）、AI 训练（GPU 间通信对丢包极敏感）、存储（NVMe-oF）。

二进制指数退避的"现代应用"

  • TCP 拥塞控制：超时后 cwnd=1，RTO 指数增长
  • HTTP 重试：1s → 2s → 4s，加 jitter 避免重试风暴
  • 微服务熔断：Hystrix、Resilience4j 默认指数退避
  • 分布式锁竞争：Redis Redlock 等待时间指数增长
  • CDN 回源重试：配合 jitter 防止同步重试
CSMA/CD 退场后，以太网如何处理"发送方过快"？
① 流控——PAUSE/PFC 暂停对端；② 拥塞控制——ECN/QCN 通知源端降速；③ 队列管理——WRED 主动丢包；④ 流量整形——Token Bucket 限速。
CSMA/CD 在交换式以太网退场，仅保留兼容性；集线器被交换机取代；同轴电缆以太网被双绞线+交换机取代；半双工以太网在主流部署淘汰；基于丢包的拥塞控制在数据中心被基于 ECN/延迟的算法（DCTCP、BBR）取代。

为什么需要 MAC 地址？有了 IP 不够吗？

IP 是网络层逻辑地址，标识主机位置（像邮寄地址，搬家就变）。MAC 是链路层物理地址，标识网卡本身（像身份证号，出厂烧录，不变）。分组在网络层用 IP 路由，但每一段链路上传输时需用 MAC 找到下一跳的具体网卡。

为什么 IP 和 MAC 都需要？只用一个不行吗？

25 交换机的自学习

为什么需要自学习

交换机连接多台设备，需知道每个 MAC 对应哪个端口才能正确转发帧。手动配置不现实（设备可能移动、增减）。自学习让交换机自动"学会"MAC 地址和端口的映射。

深度讲解

交换机自学习规则

  1. 交换机收到帧时，记录：源 MAC → 进入端口号，写入转发表
  2. 查找目的 MAC：
    • 找到 → 转发到对应端口（单播转发）
    • 找不到 → 向除源端口外的所有端口转发（泛洪 Flooding）
  3. 转发表项有老化时间（通常 300 秒），长时间不更新则删除
交换机隔离冲突域：每个端口是独立的冲突域，全双工通信，无 CSMA/CD 冲突。这是为什么现代以太网用交换机替代了集线器。

STP 协议：环路防御

自学习有致命弱点——网络环路导致广播风暴：A→B 帧被泛洪到 B，B 泛洪回 A，无限循环秒级拖垮整网。

STP（802.1D）：选举根桥，每个非根桥选到根桥最短路径端口，阻塞非最短路径端口，形成无环生成树。收敛慢 30-50 秒。

RSTP（802.1w）：收敛 1-3 秒，替代 STP 成为默认部署。

🌐 现代技术栈演进

STP 在数据中心被淘汰

STP 问题：① 收敛慢（30s）；② 阻塞链路浪费带宽；③ 不支持多路径。替代方案：VXLAN + EVPN，支持多路径（ECMP）、快速收敛、可扩展。

MLAG（多机箱链路聚合）

两台交换机虚拟成一台，下游用 LACP 连接两台交换机。全带宽利用 + 冗余。厂商方案：Cisco vPC、华为 CSS、H3C IRF、Arista MLAG。

SDN 时代的交换机

  • OpenFlow 交换机：流表代替转发表，已逐渐淡出
  • P4 可编程交换机：用户定义转发逻辑
  • 白盒交换机 + SONiC：软硬件解耦，云厂商自研标配
交换机自学习的安全风险？
① MAC 泛洪攻击——发大量不同源 MAC 撑爆 MAC 表，交换机退化为集线器泛洪所有帧（窃听）；② MAC 欺骗——伪造源 MAC 覆盖表项。防御：端口安全（限制单端口 MAC 数量）、DAI、802.1X。
VXLAN 如何解决 STP 的限制？
① 基于 UDP，可跨 L3 路由（不受 L2 STP 限制）；② ECMP 多路径；③ 24 位 VNI 支持 1600 万虚拟网络；④ EVPN 控制平面传播 MAC 避免泛洪学习。
集线器被交换机取代；传统 STP 被 RSTP 取代，数据中心被 VXLAN+EVPN 取代；Token Ring/FDDI 被以太网打败；ATM 局域网被以太网+IP 取代；纯 L2 网络在大型数据中心被 L3+Overlay 取代。

为什么需要拥塞控制

流量控制防淹没接收方，拥塞控制防淹没网络。如果所有发送方无节制地发，路由器缓冲区溢出，大量丢包，性能急剧下降——拥塞崩溃。TCP 拥塞控制让每个发送方感知网络拥塞，主动调整发送速率。

深度讲解

拥塞窗口与阈值

  • cwnd：发送方根据拥塞程度估算的窗口大小
  • ssthresh：慢开始和拥塞避免的分界点
  • 实际发送窗口 = min(cwnd, rwnd)

🎯 真题考点：路由器缓存无限大 = 拥塞吗？

真题判断："路由器缓存无限大则网络不拥塞"——错误。
原因：缓存无限大虽然不会因"丢包"判定拥塞，但数据包会在队列中无限排队，排队延迟（Queuing Delay）趋向无穷大。排队延迟飙升本身就是拥塞的表现，且会导致 TCP 超时重传，进一步恶化拥塞。
这就是 Bufferbloat（缓冲区膨胀）问题——大缓冲区掩盖了丢包信号，让拥塞控制失灵。现代方案：fq_codel / CAKE 等 AQM（主动队列管理）算法在缓冲区快满时主动丢包/打 ECN 标记，让发送方及时降速。

深度讲解

Dijkstra 算法步骤

初始化：N' = {源节点 u}；D(v) = c(u,v)（直连）或 ∞

迭代：

  1. 在 N' 之外选 D(w) 最小的节点 w，加入 N'
  2. 对 w 的所有不在 N' 中的邻居 v，更新：若 D(w)+c(w,v) < D(v)，则 D(v) = D(w)+c(w,v)，p(v) = w
  3. 重复直到所有节点都在 N' 中

最终结果：

  • u→v：距离 2，路径 u→v
  • u→w：距离 1，路径 u→w

缩写全称含义
ASAutonomous System自治系统——由单一机构管理的路由器集合（如一个运营商、一家大企业）

真题判断："OSPF is intra-AS routing protocol"——正确。

🌐 现代技术栈演进

Dijkstra 在现代路由协议的实战

OSPF：链路状态路由协议，用 Dijkstra 计算最短路径。区域分层（骨干区域 0 + 非骨干区域）避免全网泛洪。企业网、数据中心内部主流。

IS-IS：类似 OSPF 但更早（OSI 体系），扩展性更好，运营商骨干网主流。

BGP：互联网的真实路由协议

为什么互联网不用 OSPF/Dijkstra？

  • 规模：互联网百万级路由，OSPF 全网泛洪不可行
  • 策略：不同运营商有商业策略，Dijkstra 不考虑策略
  • 可扩展：BGP 支持百万路由
  • 路径向量：BGP 传完整 AS 路径避免环路

Segment Routing：下一代路由

数据包头部携带"路径指令"（Segment List），中间路由器按指令转发，无需维护路径状态。SR-MPLS 基于 MPLS，SRv6 基于 IPv6。SR 是 SDN 落地的关键技术。

OSPF 为什么要分区域？
① 减少 LSA 泛洪——同一区域内泛洪，区域间传摘要；② 降低 Dijkstra 计算量——只在区域内计算；③ 故障隔离；④ 可扩展——支持大规模网络。典型：骨干区域 0 + 多个非骨干区域，每区域 < 50 路由器。
BGP 路由泄露事件如何避免？
① RPKI（资源公钥基础设施）——基于密码学的路由源验证；② BGPSEC——BGP 路由签名；③ IRR（Internet Routing Registry）。巴基斯坦 YouTube 事件（2008）是经典案例。
RIP 被 OSPF/IS-IS 取代；EGP 被 BGP-4 取代；传统 LDP 在 SR-MPLS 部署后逐步被取代；纯最短路径路由在大型网络被 SR + SDN 流量工程取代。

用户反馈"页面打开慢"，前端如何排查？
① Chrome DevTools Network 面板看每个请求 Timing，定位 DNS/TCP/TLS/TTFB/Download 哪个慢；② Lighthouse 跑性能审计；③ Performance 面板看主线程任务；④ RUM 数据看真实用户性能；⑤ 后端追踪 TTFB 慢的原因。大厂前端面试必考。
现代 Web 性能优化的"黄金法则"是什么？
① 减少 RTT——HTTP/2、QUIC、连接预热；② 减少数据量——压缩、Code Splitting、Tree Shaking；③ 并行化——多路复用、Preload；④ 就近访问——CDN、边缘计算；⑤ 缓存复用——HTTP 缓存、Service Worker；⑥ 渐进式渲染——SSR、流式渲染、骨架屏。性能优化是系统化工程，不是单点优化。
同步 XHR 被 Fetch API 取代；Long Polling 被 WebSocket/SSE 取代；Flash/Java Applet/Silverlight 被 HTML5 取代；jQuery DOM 操作在现代框架中减少；图片 sprite 被 HTTP/2 多路复用取代；静态资源合并在 HTTP/2 时代不再必要；HTTP/1.x 优化技巧被废弃。
