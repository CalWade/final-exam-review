一、整体认知:编译原理到底在研究什么

1.1 一个最朴素的问题

你写了一行 a = b + c * 2,CPU 只认识 mov / add / imul 这种指令。中间发生了什么?

这就是编译器要解决的事。但编译原理的真正价值,远不止"写一个编译器"。它本质上是如何把一个高维抽象(人类语言)逐层降维到机器可执行的物理表示(机器码)的方法论。这套方法论渗透在现代技术栈的方方面面:

二、第一章 绪论

2.1 编译程序 vs 解释程序 vs 混合模式

类型代表语言工作方式执行速度启动速度调试
编译型C/C++/Rust/Go源码整体翻译成机器码极快慢难
解释型Python/Ruby/JS 早期逐行翻译执行慢极快容易
混合型Java/C#/JS(V8)源码→字节码→JIT→机器码快中等中等
📜 渊源
早期计算机内存稀缺,只能整块编译(FORTRAN 1957)。Lisp(1958) 开创解释执行,带来交互式 REPL。90 年代 Java 提出"一次编写,处处运行",把编译拆成"源码 → 字节码 → JVM 解释/JIT"两段,既跨平台又有性能。现代 V8 更进一步:Ignition 解释器 + TurboFan JIT + Sparkplug 优化编译器,多层编译。
🎤 面试考点
为什么 Java 叫"半编译半解释"?——因为 javac 把源码编译成字节码,但字节码由 JVM 解释执行(或 JIT 热点编译)。

2.2 编译过程与编译程序结构必背五阶段

编译过程从源程序到目标代码,经过五个阶段,每个阶段有明确的输入输出和任务:

阶段任务输入 → 输出现代对应
① 词法分析识别单词符号,过滤空格注释,输出 Token 流;同时建立符号表源程序字符流 → Token 流Scanner / Lexer
② 语法分析根据文法规则,把 Token 串归约为语法单位,生成语法树/AST;检查语法错误Token 流 → AST / 语法树Parser
③ 语义分析 + 中间代码生成类型检查、作用域检查、控制流检查;将 AST 翻译成中间代码(四元式/三地址码)AST → 中间代码 IRSemantic Analyzer
④ 代码优化对中间代码进行等价变换:局部优化(DAG)、循环优化、全局优化中间代码 → 优化后的中间代码Optimization Pass
⑤ 目标代码生成把优化后的中间代码翻译为目标机器指令;寄存器分配、指令调度优化后的 IR → 目标代码Code Generator
💡 编译程序结构 = 五阶段 + 两大配套
五个阶段构成前端 + 中端 + 后端的三段式结构:
  • 前端(Front-end):词法分析 + 语法分析 + 语义分析,与源语言相关,与目标机器无关
  • 中端(Middle-end):中间代码优化,与源语言和目标机器都无关(IR 层)
  • 后端(Back-end):目标代码生成,与目标机器相关
LLVM 是这种结构的最典型实现:Clang(前端)→ LLVM IR(中端)→ x86/ARM/RISC-V 后端。

两大配套组件(贯穿全程)

组件作用业务映射
符号表(Symbol Table)记录所有标识符的类型、作用域、内存位置;贯穿编译全过程,所有阶段共享IDE 的"跳转定义"、"自动补全"本质就是查符号表;LSP 协议;MCP 工具注册表(LLM 时代)
错误处理(Error Handler)词法错(非法字符)、语法错(缺分号)、语义错(类型不匹配);现代编译器追求"错误恢复",尽量一次性报多个错Rust/TS 友好错误信息;Babel 错误恢复;Agent Self-Correction(LLM 时代)
⚠️ 错误处理的两种策略
  • 错误恢复:尽可能继续编译,一次发现多个错误(主流策略)
  • 错误校正:尝试自动修正错误(实际很少做,容易误导用户)
详见第八章错误处理章节。

2.3 遍(pass)的概念

"一遍"指从头到尾扫描一次源码(或中间代码)。一遍编译器把多个阶段交织在一起(典型如 Pascal 早期编译器),内存友好但难优化。多遍编译器(LLVM)把每遍独立,可并行、可复用。

💡 架构师视角
遍的本质就是"数据流水线"。MapReduce、Spark、Flink 都是这种模型。编译器的"中间表示 IR"对应大数据的"DataFrame/DataSet",都是为了在阶段间传递不可变数据。

3.1.1 单词符号的分类与输出形式

单词符号(Token) 是源程序中具有独立意义的最小语法单位。分为五大类:

3.2.1 为什么需要正规式?

3.2.2 正规式(正则表达式)基础

运算符(优先级从低到高):|(或)、·(连接,可省略)、*(闭包 Kleene star)、+(正闭包)、?(可选)、() 分组。

3.2.3 有限自动机(FA)

定义:五元组 M = (Q, Σ, δ, q0, F)

  • Q:有限状态集
  • Σ:输入字母表
  • δ:转移函数
  • q0:初态
  • F:终态集
维度DFANFA
转移函数δ(q,a) = 一个状态δ(q,a) = 状态集合(可空)
ε 转移不允许允许(空字符跳转)
状态数多(通常)少(通常)
匹配速度O(n),无需回溯最坏 O(2^n),需回溯
实现复杂度跳转表,简单跳转表+回溯,复杂
⚠️ 重要定理(必背)
  • L(NFA) = L(DFA):NFA 和 DFA 表达能力相同,都识别正规语言(3 型语言)
  • NFA → DFA 的状态数最坏 2^n(指数爆炸,实际很少触发)

3.3 NFA → DFA:NFA 的确定化子集构造法 必考

3.3.1 一个核心公式

🔑 全节就记住这一句

对 DFA 的某个状态集合 I,输入字符 a 后到达的新状态:

Ia = ε-closure( move(I, a) )

拆开看两步:

  • move(I, a):I 中所有状态吃一个字符 a 能直接到的状态集
  • ε-closure(...):把上面结果沿 ε 边再扩散出去(连同自己)

就这样,反复套这个公式,把 DFA 一个状态一个状态地"打"出来。

3.3.2 打表法五步走

📐 执行步骤(打表法)
1
找起点:求 NFA 初态 q0 的 ε-闭包,作为 DFA 的初始状态(设为 A)。
2
填表格:
  • 取出一个未处理的 DFA 状态(比如 A)
  • 对于字母表里的每一个字符(比如 a 和 b),分别套上面的核心公式
  • 如果算出来的新集合以前没出现过,就给它起个新名字(B、C...),加入待处理队列
3
循环:重复步骤 2,直到没有新集合产生为止。
4
找终点:看看 A、B、C... 这些集合里,只要包含原 NFA 终态的,统统标记为 DFA 的接受状态(终态)。
5
整理:把所有状态和转移画成一张表,就是 DFA 的状态转换表。
⚠️ 两个易错点
  1. ε-closure 一定包含起点本身(不能漏)
  2. 每出现一个新状态,都要对每个字符算一遍,不能漏字符

3.3.3 完整示例必练

题目:已知正规式 (a|b)*abb 的 NFA(状态 0~10,初态 0,终态 10),求等价 DFA。

📝 打表过程

第 1 步 找起点:

A = ε-closure(0) = {0,1,2,4,7}

第 2-3 步 填表 + 循环:对每个新状态,套公式 Ia = ε-closure(move(I, a))

处理 A:
  A 看字符 a: move(A, a) = {3,8},  ε-闭包 = {1,2,3,4,6,7,8} → 新状态,命名 B
  A 看字符 b: move(A, b) = {5},    ε-闭包 = {1,2,4,5,6,7}   → 新状态,命名 C

  B 看字符 a: ε-闭包(move(B, a)) = {1,2,3,4,6,7,8} = B (老状态,自环)
  B 看字符 b: move(B, b) = {5,9}, ε-闭包 = {1,2,4,5,6,7,9} → 新状态,命名 D

处理 C:
  C 看字符 a: 同 B
  C 看字符 b: ε-闭包(move(C, b)) = {1,2,4,5,6,7} = C (自环)

处理 D:
  D 看字符 a: 同 B
  D 看字符 b: move(D, b) = {5,9,10}, ε-闭包 = {1,2,4,5,6,7,9,10} → 新状态,命名 E

处理 E:
  E 看字符 a: 同 B
  E 看字符 b: 同 C

无新状态产生,结束。

第 4 步 找终点:原 NFA 终态是 10,看哪个集合含 10?

→ A{0,1,2,4,7}BC

→ 表示初态,* 表示终态。这张表就是最终的 DFA!

3.4 DFA 的化简Hopcroft 最小化算法

3.4.1 算法思想

🔑 核心:等价状态合并
两个状态等价,当且仅当:(1) 都是终态或都是非终态 + (2) 对所有输入字符,转移到的状态也等价。
⚙️ Hopcroft 算法
1. 划分 Π = {终态集, 非终态集}
2. repeat:
     Π_new = ∅
     for 每个组 G ∈ Π:
         把 G 拆成子组:子组内每个状态对每个输入字符都转移到同一个组
         Π_new += 所有子组
     Π = Π_new
  until Π 不再变化
3. 每组选一个代表状态,合并等价状态

3.4.2 上例最小化

初始划分:Π = {{I0,I1,I2,I3}, {I4}}(终态只有 I4)

🔄 迭代过程
第一轮:
- 看 {I0,I1,I2,I3}:
  - 输入 a:都到 I1(同组)→ 不分裂
  - 输入 b:I0→I2, I1→I3, I2→I2, I3→I4,转移到不同组! → 按 b 的转移目标分组

    - I1(b→I3):单独
    - I3(b→I4):单独(因为 I4 在另一组)
- 拆成 {I0,I2}、{I1}、{I3},加上 {I4}

- 其他组已单元素

最终 4 个状态:{I0,I2}、{I1}、{I3}、{I4}

3.8 面试高频题

Q1:正则表达式引擎为什么会有 ReDoS(正则灾难)?
A:NFA 回溯引擎在嵌套量词((a+)+b)、重叠分支上会指数级回溯。可以用 RE2(DFA 引擎)避免,或者用原子组 (?>...)、占有量词 a++ 减少回溯。

Q2:为什么 DFA 比 NFA 快但状态数多?
A:DFA 每个状态对每个字符只有一条出路,无需回溯,O(n);但 NFA → DFA 最坏指数爆炸(2^n)。RE2 的策略是惰性构造 DFA,只对真正用到的子集构造。

4.1 文法和语言基础概念

4.1.1 基本概念

  • 字母表 Σ:有限符号集合,如 {a, b}
  • 符号串:字母表上的符号序列,如 abab
  • 空串 ε:长度为 0 的串
  • Σ*:所有有限符号串(含 ε)的集合
  • 语言 L:Σ* 的子集

文法 G 是一个四元组:G = (V, T, P, S)

符号含义
V非终结符集合(Non-terminal,大写字母)
T终结符集合(Terminal,小写字母/符号)
P产生式集合,形如 A → α
S开始符号(必须属于 V)

4.1.2 推导(Derivation)

符号含义
⇒直接推导(用产生式替换一个非终结符)
⇒+至少一步推导
⇒*0 步或多步推导
最左推导每步替换最左非终结符(自顶向下分析用)
最右推导每步替换最右非终结符(自底向上分析用,又叫规范推导)

4.1.3 Chomsky 文法分类必考选择题

类型别名产生式限制对应自动机代表语言
0 型短语结构 PSGα → β(α 含非终结符)图灵机任意可计算
1 型上下文有关 CSGαAβ → αγβ(γ≠ε)线性有界自动机 LBA部分自然语言
2 型上下文无关 CFGA → γ下推自动机 PDA程序语言语法
3 型正规 RGA → aB 或 A → a有限自动机 FA程序语言词法
🔑 关键结论
  • 3 型 ⊂ 2 型 ⊂ 1 型 ⊂ 0 型(层层包含)
  • 程序语言的词法用 3 型(正则)描述,语法用 2 型(CFG)描述
  • 为什么不用 1 型?太复杂,且 2 型已足够。为什么不用 3 型描述语法?因为正规文法不能表达递归嵌套结构(如括号配对)。
🤖 LLM 视角
这就是为什么 SQL 注入不能用正则防——SQL 语法是 CFG,正则(3 型)表达不了嵌套子查询。要防注入必须用 SQL Parser 把 SQL 解析成 AST 再检查。JSON 解析器也必须用 CFG。

4.1.4 正规式与上下文无关文法的关系

正规式 = 3 型文法 = NFA/DFA。但有些结构 CFG 能表达、正规式不能:

📚 经典反例:括号配对 L = {aⁿbⁿ | n ≥ 1}
  • CFG 表达:S → aSb | ab
  • 正规式:无法表达(自动机没有"无限计数"能力)

4.2.2 短语、直接短语、句柄必考概念

📐 定义

设 S ⇒* αAβ ⇒+ αγβ(γ 至少一步推导出),则:

  • 短语(Phrase):γ 是句型 αγβ 相对于 A 的短语
    直观:语法树中每棵子树的叶子序列
  • 直接短语(Direct Phrase):S ⇒* αAβ ⇒ αγβ(一步推导)
    直观:只有父子两层的子树的叶子序列
  • 句柄(Handle):最左直接短语(自底向上分析的关键)

  • 三个叶子:i、i、i

直接短语(只有两层):i、i、i(三个叶子)

句柄(最左直接短语):i(第一个 i)

🚫 二义性(Ambiguity)
如果一个文法存在某个句子有两棵不同的语法树,则文法二义。

E → E+T | T       (低优先级)
T → T*F | F       (高优先级)
F → (E) | i       (最高,原子)

4.3 自顶向下语法分析:LL(1) 分析法综合题第 3 题 15 分

4.3.1 自顶向下分析的思想

从开始符号 S 出发,最左推导,尝试推出输入串。问题:每步推导可能有多条产生式可选,选错了要回溯。

💡 LL(1) 含义
  • 第一个 L:从左到右扫描输入
  • 第二个 L:最左推导
  • 1:向前看 1 个符号决定产生式

4.3.2 三个核心问题

要无回溯,文法必须满足:
  1. 无左递归:A → Aα 会导致死循环
  2. 无回溯:每步选择产生式唯一确定
  3. 无公共左因子:A → αβ | αγ 需要提取左因子

4.3.3 消除左递归必考

📐 直接左递归消除
原: A → Aα | β
改: A → βA'
     A' → αA' | ε

例:E → E+T | T 改为 E → TE',E' → +TE' | ε

4.3.4 提取左因子

原: A → αβ1 | αβ2
改: A → αA'
     A' → β1 | β2

4.3.5 FIRST 集必考,15 分核心

定义:FIRST(α) = {a | α ⇒* a..., a ∈ T} ∪ (若 α ⇒* ε,则加 ε)

直观理解:α 能推出的所有"开头终结符"集合。

⚙️ FIRST 集计算算法
  1. 若 X ∈ T:FIRST(X) = {X}
  2. 若 X → ε 是产生式:加 ε
  3. 若 X → Y1Y2...Yk:
    • 把 FIRST(Y1) \ {ε} 加入 FIRST(X)
    • 若 ε ∈ FIRST(Y1),把 FIRST(Y2) \ {ε} 加入;以此类推
    • 若所有 Yi 都能推 ε,加 ε

例:文法

E  → TE'
E' → +TE' | ε
T  → FT'
T' → *FT' | ε
F  → (E) | i
非终结符FIRST 集
F{ (, i }
T'{ *, ε }
T{ (, i }(F 不能推 ε)
E'{ +, ε }
E{ (, i }(T 不能推 ε)

4.3.6 FOLLOW 集必考

定义:FOLLOW(A) = {a | S ⇒* ...Aa..., a ∈ T} ∪ (若 A 是最右符号,加 #)

直观理解:在某个句型中,紧跟在 A 后面的终结符。

⚙️ FOLLOW 集计算算法
  1. # ∈ FOLLOW(S)(S 是开始符号)
  2. 对产生式 A → αBβ:
    • 把 FIRST(β) \ {ε} 加入 FOLLOW(B)
    • 若 ε ∈ FIRST(β)(即 β 可推空),把 FOLLOW(A) 加入 FOLLOW(B)
  3. 重复直到不再变化

例(同上文法):

非终结符FOLLOW 集推导
E{ #, ) }F → (E),E 后跟 ); S=E 加 #
E'{ #, ) }E' 在 E → TE' 末尾,继承 FOLLOW(E)
T{ +, #, ) }FIRST(E')\ε ∪ FOLLOW(E)
T'{ +, #, ) }T' 在 T → FT' 末尾,继承 FOLLOW(T)
F{ *, +, #, ) }FIRST(T')\ε ∪ FOLLOW(T)
⚠️ 永远不含 ε
FOLLOW(A) 中永远不含 ε,只可能有终结符和 #!这是高频考点。

4.3.7 LL(1) 分析表构造15 分大题终点

📐 构造规则

对每个产生式 A → α:

  1. 对 FIRST(α) 中每个终结符 a:M[A, a] = A → α
  2. 若 ε ∈ FIRST(α):对 FOLLOW(A) 中每个 b:M[A, b] = A → α

例:

E'E'→+TE'E'→εE'→ε

⚙️ 算法

初始:# 入栈,S 入栈,输入指针指首字符

  • 若 X = a = #:接受

    • 无产生式:报错

4.4 自底向上语法分析:LR 分析法综合题第 4 题 12 分

4.4.1 自底向上分析的思想

从输入串开始,最左归约(最右推导的逆),目标是归约到 S。归约(Reduce):产生式右部 → 左部,是推导的逆操作。

例:i+i*i 归约过程:

i+i*i  → F+i*i  → T+i*i  → E+i*i  → E+F*i  → E+T*i  → E+T  → E

4.4.2 LR 分析的家族

文法向前看能力难点
LR(0)0弱状态多,冲突多
SLR(1)1中用 FOLLOW 解决冲突
LALR(1)1中强合并同心状态,Yacc 默认
LR(1)1强状态爆炸

能力:LR(0) < SLR(1) < LALR(1) < LR(1)

💼 业务场景
Yacc/Bison 默认用 LALR(1),GCC 早期用 Bison,后改为手写递归下降(性能和错误恢复)。PostgreSQL SQL parser 也是 Bison + LALR(1)。

4.4.3 LR(0) 项目与项目集

LR(0) 项目:产生式右部加一个点 ·,表示分析位置。

项目形式含义
A → ·αβ还没看任何字符
A → α·β看了 α
A → αβ·归约项目(可以归约)
⚙️ 项目集闭包 closure(I)
  1. I 中所有项目加入
  2. 若 A → α·Bβ ∈ I(B 是非终结符),对 B 的所有产生式 B → γ,加入 B → ·γ
  3. 重复直到不变

GOTO 函数:GOTO(I, X) = closure({A → αX·β | A → α·Xβ ∈ I})

4.4.4 拓广文法必考第一步

🎯 目的
让分析器有一个唯一的接受状态。
做法:新增开始符号 S' 和产生式 S' → S。

例:原文法 S → aSb | ab,拓广为:

(0) S' → S
(1) S → aSb
(2) S → ab

4.4.5 构造识别活前缀的 DFA综合题核心

🔑 活前缀(Viable Prefix)
规范归约中,栈里始终是某规范句型的前缀,且不超过句柄右端。LR 分析就是要识别所有活前缀。
⚙️ 算法
  1. 初态 I0 = closure({S' → ·S})
  2. 对每个项目集中的每个 · 后符号 X,计算 GOTO(I, X)
  3. 把新项目集加入,直到不变
  4. 状态间用 X 标号的边连接

4.4.7 LR(0) 文法的冲突

⚠️ LR(0) 要求
每个状态不能同时:
  • 既有移进又有归约(移进-归约冲突)
  • 有多个归约(归约-归约冲突)

⚙️ 算法

初始:状态 0 入栈,输入指针指首字符。

  • sj:状态 j 入栈,指针前进(移进)
  • rj:用第 j 条产生式 A → α 归约:弹出 |α| 个状态,看新栈顶 s' 和 A,把 GOTO[s', A] 入栈
  • acc:接受
  • 空:报错

4.7 面试高频题

Q1:为什么 LR 比 LL 强?
A:LL 在推导时只看当前非终结符和向前看符号,选择产生式;LR 在归约时已经"看到"了完整右部,信息更充分。LR 能处理左递归,LL 不能。

Q2:Bison 默认用 LALR(1) 而非 LR(1),为什么?
A:LR(1) 状态数最坏指数爆炸,LALR(1) 合并同心状态,状态数与 SLR 相同,但能力接近 LR(1)。代价:可能引入归约-归约冲突,但实际很少触发。

Q3:手写递归下降和 Yacc 生成的 parser,该选哪个?
A:现代主流编译器(GCC、Clang、V8、TypeScript、Rust)都是手写递归下降,因为性能更好、错误恢复更灵活、错误信息更友好。Yacc/Bison 主要用于 DSL、配置文件、SQL parser 这种规则相对简单的场景。

5.5 表达式及赋值语句的翻译

5.5.1 简单算术表达式的翻译

产生式           语义动作(生成四元式)
E → E1 + E2     { E.place := newtemp;
                  emit('+', E1.place, E2.place, E.place) }
E → E1 * E2     { E.place := newtemp;
                  emit('*', E1.place, E2.place, E.place) }

newtemp:分配临时变量 t1, t2, ...;emit:输出一条四元式。

两种翻译方式:

方式说明

真链(true list) 和 假链(false list):记录所有"为真时该跳到哪"和"为假时该跳到哪"的四元式位置,等目标地址确定后回填(backpatching)。

例:a or b 的翻译:

(2) (j, -, -, 3)         // a 假,顺序执行 b
(3) (jnz, b, -, 0)      // b 真,跳到真出口(待回填)
(4) (j, -, -, 0)         // b 假,跳到假出口(待回填)

5.6.3 switch 语句的翻译

两种实现:

  1. 条件跳转链:每个 case 顺序比较(适合 case 少)
  2. 跳转表(jump table):把 case 值和目标地址做成表,直接索引(适合 case 多且密集,C 编译器常用)
💼 业务场景
C 编译器对 switch-case 优化:case 少用条件链,case 多且密集用跳转表,稀疏用 hash 表。

5.6.4 数组元素的翻译

📐 数组地址计算

一维 A[i](下标从 0,元素宽度 w):addr(A[i]) = base(A) + i * w

二维 A[i][j](行优先,行大小 n2*w):addr(A[i][j]) = base(A) + (i * n2 + j) * w

数组元素赋值的四元式:

x := A[i]      // 取下标
([], A, i, t)        // t := A[i]
(:=, t, -, x)        // x := t

A[i] := x      // 存下标
([]=, x, -, A[i])    // A[i] := x

5.6.5 过程调用的翻译

参数传递:用 param 四元式压参,最后 call。

例:f(a, b+c, d):

(1) (+, b, c, t1)       // 算 b+c
(2) (param, a, -, -)    // 压第 1 参
(3) (param, t1, -, -)   // 压第 2 参
(4) (param, d, -, -)    // 压第 3 参

6.2 局部优化:基本块与 DAG

6.2.1 基本块(Basic Block)的划分

📐 定义
顺序执行的指令序列,只有一个入口(第一条)和一个出口(最后一条)。
⚙️ 划分算法
  1. 找入口(Leaders):
    • 程序第一条指令
    • 跳转目标(条件/无条件跳转的目标地址)
    • 跳转指令的下一条(条件跳转的下一条)
  2. 划分:每个 leader 到下一个 leader 前 = 一个基本块

例:

(1)  read x
(2)  if x > 0 goto (5)
(3)  x := -x
(4)  goto (5)
(5)  y := x + 1

Leaders: (1)、(3) [(2)条件跳转的下一条]、(5) [(2)和(4)的跳转目标]

基本块指令
B1(1)-(2)
B2(3)-(4)
B3(5)

6.2.2 程序流图(CFG)

💼 业务场景
LLVM 的 Function 由 BasicBlock 组成,CFG 是所有优化 Pass 的工作基础。PProf、火焰图、perf 都依赖 CFG。

6.2.3 基本块的 DAG 构造必练

DAG 节点类型:

  • 叶节点:变量常量(标记为标识符),带初值
  • 内部节点:运算符,带操作数子节点,标记为运算结果变量
⚙️ 构造算法(对基本块每条四元式)
  1. (:=, a, -, b)(b := a):
    • 找/建 a 的节点
    • 把 b 加到该节点的"附加标识符"集合
  2. (op, a, b, c)(c := op a b):
    • 找/建 a、b 的节点
    • 查找已存在的"以 op 为标记,a 和 b 为子节点"的节点
      • 若存在,c 加到该节点附加标识符集(公共子表达式复用)
      • 若不存在,新建节点,c 加附加标识符
  3. ([]=, ...) 数组赋值要特殊处理(杀掉所有依赖于数组的节点)

6.2.4 DAG 优化示例必练

原四元式序列:

(1) (+, a, b, t1)     // t1 := a + b
(2) (+, a, b, t2)     // t2 := a + b  (公共子表达式)
(3) (*, t1, t2, t3)   // t3 := t1 * t2
(4) (:=, t3, -, c)    // c := t3

DAG 构造过程:

  1. (1):建叶 a、b,建节点 n1(+, a, b),t1 附加
  2. (2):找已有 (+, a, b) 节点 = n1,t2 附加到 n1(复用!)
  3. (3):建节点 n2(*, n1, n1),t3 附加
  4. (4):c 附加到 n2
图 6-1 · DAG 示例:t1 和 t2 复用同一个 + 节点(公共子表达式消除)

从 DAG 重新生成四元式:

(1) (+, a, b, t1)     // t1 := a + b
(2) (*, t1, t1, t3)   // t3 := t1 * t1  (t2 复用 t1)
(3) (:=, t3, -, c)    // c := t3

优化效果:

  • 公共子表达式消除(a+b 只算一次)
  • 死代码消除(t2 被消除)
  • 临时变量复用

6.3 循环优化选择/简答可能考

6.3.1 循环的定义与查找

📐 关键概念
  • 必经节点(Dominator):从入口到节点 n 的所有路径都经过 d,则 d 支配 n
  • 回边(Back Edge):边 n → d,其中 d 支配 n
  • 自然循环:由回边 n → d 确定的循环 = {n, d} + d 到 n 的所有路径上的节点

6.3.2 三大循环优化

① 代码外提(Loop-Invariant Code Motion)
思想:循环不变的计算提到循环外。
例: 
while (i < n) {
    a = b * c + 10;   // b、c 不变
    i++;
}
→ 优化为:
t = b * c + 10;
while (i < n) {
    a = t;
    i++;
}
② 强度削弱(Strength Reduction)
思想:用便宜运算替代贵的运算。
例:x * 2 → x + x 或 x << 1;循环内 i * 4 → 维护 j,每次 j += 4
③ 归纳变量删除(Induction Variable Elimination)
思想:循环里多个归纳变量(同步变化)只保留一个。
例: 
for (i = 0; i < n; i++) {
    j = i * 4;    // j 是 i 的归纳变量
    a[j] = 0;
}
→ 优化为:
for (j = 0; j < 4*n; j += 4) {
    a[j] = 0;
}
// i 被删除(若循环外不用)
🤖 现代编译器对应 Pass
  • LLVM:LoopStrengthReduce、LICM、IndVarSimplify
  • JVM C2:Sea of Nodes 上的循环优化
  • LLM 时代:Prefix Caching / KV Cache 复用,本质是循环不变量外提

6.4.1 数据流分析(Data-Flow Analysis)

到达-定值(Reaching Definition):变量在某点"被定义"后,到另一点是否未被覆盖。

⚙️ 数据流方程
  • gen[B]:基本块内新产生的定值
  • kill[B]:基本块内被覆盖的定值
  • in[B] = ∪ out[P],P ∈ 前驱
  • out[B] = gen[B] ∪ (in[B] - kill[B])
🤖 业务场景
IDE 的"未使用变量警告"、SQL 优化器的列依赖分析,都是数据流分析的应用。LLVM SSA 中每个 use 直接指向唯一定义,数据流分析天然显式。

6.4.2 du 链与 ud 链

链类型方向说明
ud 链(Use-Definition)使用 → 定义某变量使用点 → 所有定义点
du 链(Definition-Use)定义 → 使用某定义点 → 所有使用点
🤖 业务场景
LLVM SSA 中每个 use 直接指向唯一定义,ud/du 链天然显式。这是 SSA 优于四元式的核心原因。

6.4.3 复写传播(Copy Propagation)

6.6 面试高频题

Q1:为什么 LLVM 用 SSA,而 GCC 早期不用?
A:SSA 显式表达 def-use 链,使数据流分析算法大幅简化,优化 Pass 写起来更简单。GCC 4.0(2005)之后才引入 SSA(GIMPLE),追上 LLVM。现代编译器后端几乎全 SSA。

Q2:公共子表达式消除(CSE)和 SSA 的关系?
A:SSA 下,相同变量名只赋值一次,所以"同名"等价于"同值"。但 CSE 仍需识别"运算相同且操作数相同",SSA 只是基础。

Q3:为什么循环展开能加速?
A:减少循环控制开销(条件判断、跳转)、增大指令级并行(ILP)窗口、便于 SIMD 向量化。但展开过多会增大代码体积,损害 I-cache 命中。LLVM 有 unroll heuristics 自动决定展开因子。

Q4:哪些优化会改变语义?
A:浮点运算优化最敏感:x*0 → 0 对 NaN 不成立;a+b+c 重组顺序可能改变结果。Java 用 strictfp 关键字约束,LLVM 用 fast-math 标志放宽。

❓ 问题

8.1 简单代码生成器

8.1.1 待用信息与活跃信息必懂

📐 定义
  • 待用信息(Next Use):某变量在当前点之后,下次被使用的位置
  • 活跃信息(Live):某变量在当前点之后是否还会被使用
⚙️ 计算:从基本块末尾反向扫描

对每条指令:

  1. 对结果变量 x:当前 x 的待用 = ∅、活跃 = false(此点之前不会被用)
  2. 对操作数变量 y、z:当前 y/z 的待用 = 当前指令、活跃 = true
  3. 更新 x、y、z 的待用/活跃表
🤖 业务场景
LLVM 的 LiveIntervals、RegisterCoalescer,Java HotSpot 的 Liveness Analysis,都是这个概念的工业级实现。

8.1.2 代码生成算法寄存器分配

⚙️ 简单算法(基于待用/活跃信息)

x := y op z:

  1. 取 y、z 的当前寄存器(若不在内存则加载到寄存器)
  2. 选一个空闲寄存器 R(优先选不活跃的)
  3. 生成 op R, Ry, Rz(或 op Ry, Rz, R)
  4. 更新 x → R 的映射

x := y(赋值):把 y 的寄存器也关联到 x(若 y 不再活跃)

8.1.3 寄存器分配算法

📐 图着色算法(Chaitin)
思想:把寄存器分配建模为图着色问题。
  1. 冲突图(Interference Graph):节点 = 变量,边 = 同时活跃(冲突)
  2. 着色:用 k 种颜色(k = 寄存器数)给图着色,相邻节点不同色
  3. 溢出(Spill):若不可 k-着色,把某些变量溢出到内存
复杂度:NP-hard,实际用启发式(Chaitin-Briggs、PBQP)。
📐 线性扫描(Linear Scan)
思想:按变量的活跃区间排序,一次扫描分配。
  1. 计算所有变量的活跃区间
  2. 按起点排序
  3. 扫描,用活跃集合维护"当前占用寄存器的变量"
  4. 若寄存器不够,选一个最远结束的溢出
复杂度:O(n log n),远快于图着色。
🤖 业务场景
  • GCC、LLVM 早期都用图着色。LLVM 11 之后默认用 RegAllocGreedy(贪心 + 图着色变体)
  • JVM HotSpot Client Compiler(C1)用线性扫描,追求低编译延迟。V8 也用线性扫描
  • LLM 时代:GPU 显存调度(KV Block 分配)借鉴同样的分配思想

10.1 综合题 1:NFA → DFA → 最小化12 分

🔑 一句话记忆

整个题就一个公式反复用:Ia = ε-closure(move(I, a))

分两阶段:① 子集构造(NFA → DFA)+ ② Hopcroft 最小化(DFA → 最小 DFA)。

第一阶段:子集构造(打表法)

📐 执行步骤(打表法)
1
找起点:求 NFA 初态 q0 的 ε-闭包,作为 DFA 的初始状态(设为 A)。
2
填表格:取出一个未处理的 DFA 状态(比如 A),对字母表里每个字符(a、b),分别套公式 Ia = ε-closure(move(I, a))。算出的新集合若是第一次出现,起个新名字(B、C...),加入待处理队列。
3
循环:重复步骤 2,直到没有新集合产生为止。
4
找终点:看 A, B, C... 这些集合里,只要包含原 NFA 终态的,统统标记为 DFA 的接受状态。
⚠️ 易错点
  • ε-closure 一定包含起点本身(漏写就全错)
  • 每个新状态都要对每个字符算一遍,不能漏字符
  • 终态判断:看集合里有没有 NFA 终态,不是看集合大小

第二阶段:Hopcroft 最小化

📐 执行步骤(划分法)
1
初始划分:把所有状态分成两组:Π = {终态组, 非终态组}
2
尝试拆分:对每个组,检查组内每个状态。对每个输入字符,看它们转移到哪个组:
  • 如果组内所有状态对每个字符都转移到同一个组 → 这个组不用拆
  • 如果有状态转移到不同组 → 按转移目标把这些状态拆成子组
3
循环:重复步骤 2,直到所有组都拆不动为止(Π 不再变化)。
4
合并:每组选一个代表状态,合并等价状态,重命名,得到最小 DFA。
⚠️ 最小化最大坑点

判断两个状态是否等价,看的是"转移到不同组",不是"转移到不同状态"!

举例:状态 X 和 Y,输入 a 后 X 转到 B、Y 也转到 B(同一个组),即使 X、Y 名字不同,只要转移目标在同一组,它们就可能等价。

✅ 自检技巧
最小化后,DFA 状态数应该 ≤ 原状态数。若新状态数更多,肯定错。
合理的最小 DFA 状态数通常是原 DFA 状态数的 60%~80%。
✅ 自检技巧
最小化后,DFA 状态数应该 ≤ 原状态数。若新状态数更多,肯定错。

2
画语法树:每步产生式 = 一棵子树
3
找短语:每棵子树的叶子序列(从根到叶子遍历所有子树)
4
找直接短语:只有两层的子树;句柄 = 最左直接短语
🔑 口诀
  • 短语 = 任意子树叶子
  • 直接短语 = 二层子树叶子
  • 句柄 = 最左直接短语

10.3 综合题 3:FIRST/FOLLOW + LL(1) 分析表15 分

📐 5 步标准流程
1
消除左递归 + 提取左因子(若需要)
2
求 FIRST 集(自底向上):终结符 X:FIRST(X)={X};A→Xβ:FIRST(X)\{ε} 加入;若 X 可推 ε,继续看 β 的下一个符号;全可推 ε → 加 ε
3
求 FOLLOW 集(从 S 开始迭代):#∈FOLLOW(S);A→αBβ:FIRST(β)\{ε} 加入 FOLLOW(B);A→αB(或 β 可推 ε):FOLLOW(A) 加入 FOLLOW(B);迭代到不变
4
构造分析表:对每条 A→α:对 FIRST(α) 中每个终结符 a:M[A,a]=A→α;若 ε ∈ FIRST(α):对 FOLLOW(A) 中每个 b:M[A,b]=A→α
5
判定 LL(1):每个格子最多一个产生式 → LL(1)
⚠️ 易错点 TOP5
  1. FIRST 计算漏 ε(尤其是符号串 α = X1X2...Xk,所有 Xi 都能推 ε 才加 ε)
  2. FOLLOW 迭代不充分(至少 2 轮直到不变)
  3. FOLLOW 误加 ε(FOLLOW 永远不含 ε!)
  4. 填表时,A → α 中 α 可推 ε 才用 FOLLOW 填,不能直接对 A → ε 用 FOLLOW
  5. 多个产生式填同格 → 非 LL(1),别强行只填一个

10.4 综合题 4:LR(0) 分析表12 分

📐 4 步标准流程
1
拓广文法:加 S' → S;所有产生式编号(0=S'→S,1、2、3... 依次)
2
构造项目集族(DFA 状态):初态 I0 = closure({S' → ·S});closure:A → α·Bβ 加入所有 B → ·γ;GOTO:GOTO(I, X) = closure({A → αX·β | A → α·Xβ ∈ I});直到无新状态
3
画 DFA:状态用方框,标项目集内容;边用符号(X 转移到 GOTO(I, X));同状态可自环
4
填分析表:
  • 移进:A → α·aβ ∈ Ii,GOTO(Ii, a) = Ij → ACTION[i, a] = sj
  • 归约:A → α· ∈ Ii(A ≠ S'),用第 k 条产生式 → ACTION[i, *] = rk(LR(0) 对所有终结符 + #)
  • 接受:S' → S· ∈ Ii → ACTION[i, #] = acc
  • GOTO:GOTO(Ii, A) = Ij → GOTO[i, A] = j
⚠️ 易错点 TOP5
  1. 拓广文法忘加 S' → S
  2. closure 不完整(· 后非终结符的所有产生式都要加)
  3. GOTO 漏算某个符号
  4. LR(0) 归约填表要填所有终结符 + #,不是只填 FOLLOW(那是 SLR(1))
  5. 项目集编号要连续,不能跳号
🔍 冲突检测
  • 移进-归约冲突:某状态既有 A → α·aβ 又有 B → γ·
  • 归约-归约冲突:某状态有两个归约项目
  • 有冲突 → 非 LR(0),需用 SLR(1)/LALR(1)/LR(1)

10.12 高频面试题汇总拓展

Q1:编译器和解释器的本质区别?
A:编译器一次性把源码翻译成目标代码,目标代码独立运行;解释器逐条翻译执行,不产生独立目标程序。JIT 是混合模式:运行时把热点代码编译成机器码缓存。

Q2:为什么 LLVM 是"通用编译器后端"?
A:LLVM 把"语言前端(Clang/Rust/Swift frontend)"和"目标后端(x86/ARM/RISC-V)"解耦,通过统一的 LLVM IR 中介,实现"前端复用后端、后端复用前端"的 N×M → N+M 简化。

Q3:TypeScript 的类型系统是编译期的还是运行时的?
A:编译期。TS 编译后是 JS,类型信息被擦除(Type Erasure)。这叫"语法制导的静态类型检查",对应属性文法。Java 的泛型也是类型擦除。

Q4:为什么 Rust 没有 GC 还能保证内存安全?
A:编译期借用检查器(Borrow Checker)基于数据流分析,静态验证所有权和借用规则,违反则编译失败。本质是"语义分析"的极致应用。

Q5:WebAssembly 为什么快?
A:(1) 字节码接近机器码,JIT/AOT 编译快;(2) 栈式虚拟机,无寄存器分配开销;(3) 强类型,无运行时类型检查;(4) 沙箱隔离,无副作用。本质是"中间代码 + 即时编译"。

Q6:SQL 注入为什么不能用正则防?
A:SQL 是 CFG(2 型文法),正则只能识别 3 型文法。要防注入必须用 SQL Parser 解析成 AST,检查结构和参数边界。Druid、ShardingSphere 都用这种方式。

Q7:为什么 Go 编译这么快?
A:(1) 语法简单,LL(1) 子集;(2) 无模板元编程(C++ 模板是图灵完备的编译期语言);(3) 包级别并行编译;(4) 不做激进优化,优化 Pass 少;(5) 自带依赖管理,避免重复编译。

Q8:JIT 和 AOT 的取舍?
A:JIT(Java/C#/JS)运行时编译,可基于 profile 优化,启动慢但峰值高;AOT(C/Rust/Go)编译期完成,启动快但无法用运行时信息。现代趋势是混合:Java GraalVM、V8 TurboFan、Android ART 都同时支持。

Q9:tree-shaking 和死代码消除是一回事吗?
A:概念相似,实现不同。DCE 基于数据流分析(LLVM DCE),tree-shaking 基于 ES Module 静态导入分析(Webpack/Rollup)。前端用 tree-shaking 是因为 JS 动态特性,DCE 难做。

Q10:程序语言为什么不用正则描述语法?
A:正则(3 型文法)表达能力有限,无法描述嵌套结构(括号配对、嵌套 if)。语法用 CFG(2 型文法),可由下推自动机识别,支持无限递归嵌套。

10.13 大模型相关高频面试题增量

Q11:LLM 的 Tokenizer 和编译器的词法分析器有什么关系?
A:本质都是 DFA。BPE 的合并规则构建一棵确定性合并树,推理时按规则贪心切分字节流,与 DFA 状态转移完全同构。tiktoken 用 BPE,HuggingFace tokenizers 还支持 WordPiece、Unigram,都是确定性自动机。区别:编译器 lexer 输出 Token + 属性,tokenizer 输出 token id;编译器 lexer 要识别多种 token 类型,tokenizer 只产生一种(文本片段)。

Q12:如何让 LLM 一定输出合法 JSON?
A:三层保障:(1) Prompt 工程(提示词约束);(2) Function Calling / Tool Use(JSON Schema 强约束);(3) grammar-constrained decoding(Outlines/LMQL/llama.cpp GBNF)——在采样时跑一个 LR/LL 分析器,只允许走文法可接受的 token,本质是语法分析引导生成。第三种最可靠,工业级用 vLLM + outlines 集成。

Q13:vLLM 的 PagedAttention 借鉴了什么思想?
A:操作系统的虚拟内存分页 + 编译器的存储分配思想。传统 LLM 推理 KV Cache 连续分配,显存碎片严重;PagedAttention 把 KV 切成固定大小 block,用 Block Table 索引,逻辑连续物理分散。这和编译器栈式/堆式分配管理内存的本质是一样的——把"逻辑对象"和"物理位置"解耦。

Q14:Agent 的 ReAct 和 Plan-and-Execute 哪个更像编译器?
A:Plan-and-Execute 更像编译器——先生成完整计划(AST),再分步执行(代码生成 + 运行),对应"先分析后执行"。ReAct 更像解释器——边想边做(Thought→Action→Observation 循环),每步基于上一步结果,对应"逐条解释执行"。混合模式:先 Plan 编译,执行时 ReAct 解释,失败时 Reflect 重编译,这就是 JIT 的 Agent 版。

Q15:Speculative Decoding 和 CPU 推测执行是一回事吗?
A:思想同源,实现不同。CPU 推测执行预判分支,验证成功则继续,失败则回滚流水线;LLM Speculative Decoding 用小模型先并行生成 k 个 token,大模型批量验证,命中即跳过大模型的串行解码。本质都是"用便宜计算代替贵计算 + 失败回退"。

Q16:MCP 协议和编译器符号表有什么关系?
A:MCP(Model Context Protocol)本质是 Agent 的"符号表协议"。MCP Server 暴露工具/资源/Prompt 的元数据(name、schema、description),Client(LLM)通过 MCP 协议发现和查询工具,等价于编译器在符号表中查询标识符的类型、作用域、签名。没有 MCP 之前,每个 Agent 框架(LangChain/AutoGPT/Claude)各自维护工具注册表,等价于没有统一 ABI 的"野编译器"。

Q17:LangGraph 和编译器 CFG 有什么相通之处?
A:LangGraph 的 StateGraph 节点是 Agent 状态,边是状态转移,本质就是带数据流的状态机。编译器的 CFG(Control Flow Graph)描述程序基本块间控制流,数据流分析(到达-定值、活跃变量)用于优化。LangGraph 的 Checkpointer 等价于 SSA 的版本化状态,可回溯、可分支。Agent 的循环检测(避免死循环)等价于编译器的循环不变量分析。
